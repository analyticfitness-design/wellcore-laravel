**Estado**: ⏸️ **PAUSADO 2026-05-17** — Fase 0 completa y decisiones de producto resueltas (ver §8). Daniel retomará después de cerrar el motor v2 (en desarrollo). Cuando reanude, arrancamos **plan completo (Fases 1-5)**.

## 0. Decisiones de producto resueltas por Daniel (2026-05-17)

1. **No se construye UI de checkout/billing en USD** en este proyecto — solo infraestructura técnica de locale. La decisión USD vs COP en pagos al cliente queda diferida.
2. **Dashboard COACH muestra cifras monetarias en USD para coaches en-US** (revenue, MRR, comisiones) porque los coaches estadounidenses cobran en USD por su trabajo. Implementación: agregar columna `admins.currency VARCHAR(3) NOT NULL DEFAULT 'COP'`; derivar default al crear (`locale='en'`→`'USD'`, `locale='es'`→`'COP'`); editable manualmente desde panel admin.
3. **Rutas vanilla PHP NO se tocan ni se migran** previamente. Daniel acepta como deuda técnica conocida que un cliente en-US navegando a una ruta vanilla verá copy en español. El modo de falla #1 original queda registrado como deuda asumida, no como bloqueador.
4. **Coach asignado a cliente US debe ser native English speaker** (US o de otros países anglos). No se exige bilingüe. Implementación implícita: el coach tiene `admins.locale='en'` + `admins.currency='USD'`. No se necesita columna adicional como `languages_spoken`. La asignación coach↔cliente se valida en la lógica de matching: si `client.locale='en'`, sólo coaches con `locale='en'` son elegibles.

**Estas decisiones se aplicarán en Fase 1 al arrancar.**

  - `resources/lang/` (con sólo `es/` + backups) — **directorio legacy**, ignorar.

- **Estimación total**: **18–26 jornadas Claude-asistidas** (~3-4 semanas calendario).
- **Modelo de coach**: vive en `app/Models/Admin.php` con `role`. No hay `Coach.php`. Esto simplifica: una sola tabla `admins` para locale en backoffice.
- **Render**: 100% CSR Vue 3 SPA para dashboards. Solo `/login` y `<title>` son SSR/Blade. **Implicación**: el grueso del trabajo está en Vue, no en Blade.

## 4. Decisiones arquitecturales (con confianza)

### Decisión 1 — Stack i18n: **Laravel `lang/` + `vue-i18n@9` + endpoint API híbrido** | Confianza ALTA

- Mantener `lang/es/` y `lang/en/` existentes (espina dorsal Blade/Livewire/Mail).
- Instalar `vue-i18n@9` para frontend (verificar `package.json` antes; agregar manualmente y `npm install` sin args — regla supply-chain CLAUDE.md).
- Endpoint nuevo: `GET /api/v/translations/{locale}.json` sirviendo namespace por rol (cliente vs coach). Cache HTTP largo + versionado por hash de archivos. Cache Redis 1h.
- **Crítico**: el primer paint inyecta `<meta name="wc-locale">` + `window.__wcMessages` con los namespaces mínimos (nav, dashboard, common) desde Blade root → **0 FOUC**.

**Trade-off explícito**: app vanilla PHP NO se toca (ADR-0001). Si un cliente en-US navega a una ruta vanilla, ve español. Mitigación: §8 ambigüedad 4.

### Decisión 2 — Persistencia locale + unidades | Confianza ALTA

Migraciones aditivas (cumplen ADR-0003):

```sql
ALTER TABLE clients
  ADD COLUMN locale VARCHAR(10) NOT NULL DEFAULT 'es' AFTER timezone,
  ADD COLUMN locale_locked TINYINT(1) NOT NULL DEFAULT 0 AFTER locale,
  ADD COLUMN unit_system VARCHAR(10) NOT NULL DEFAULT 'metric' AFTER locale_locked;

ALTER TABLE admins
  ADD COLUMN locale VARCHAR(10) NOT NULL DEFAULT 'es' AFTER email,
  ADD COLUMN locale_locked TINYINT(1) NOT NULL DEFAULT 0 AFTER locale;

ALTER TABLE assigned_plans
  ADD COLUMN plan_data_en JSON NULL AFTER plan_data;
```

**Justificaciones**:
- `VARCHAR(10)` NO ENUM: lección del incidente real `clients.plan` ENUM-en-prod (2026-04). VARCHAR deja agregar `pt-BR`, `es-MX`, `en-GB` sin migrar tipo. Validación en FormRequest, no en DB.
- Valores `'es'`/`'en'` (no `'es-CO'`/`'en-US'`): alinear con código actual y `SetLocale` middleware. La voz "Colombia neutro" vs "US native" es responsabilidad de copy, no de code de locale.
- Coach va en `admins`: porque NO existe tabla `coaches`. Daniel (superadmin) tendrá `locale_locked=1` desde seed.
- `unit_system` separado del `locale`: cliente en-US fitness-nerd puede preferir kg; cliente es-MX entrenando con coach US puede querer lbs. Default derivado del locale (`es→metric`, `en→imperial`) editable.

### Decisión 3 — Toggle + lock + sincronización | Confianza MEDIA-ALTA

- Composable Vue `useLocale()`: lee `window.__wcInitialLocale` (inyectado por Blade) → previene FOUC. Persistencia: `PATCH /api/v/me/locale` + Pinia store + cookie `wc_locale` (1 año, SameSite=Lax).
- Lock: backend rechaza 403 si `locale_locked=true` y request no viene de un admin. Frontend deshabilita el toggle con tooltip ("Language set by your coach team" / "Idioma asignado por tu coach").
- Multi-tab: cookie + columna como fuente de verdad. Cambio: escribir columna primero → Set-Cookie en respuesta. Para sync inmediato entre tabs: opcional broadcast Reverb `LocaleChanged` (puede diferirse a v1.1).

### Decisión 4 — Contenido motor v2: **bilingüe al PERSIST** (Opción C) | Confianza ALTA

- Motor v2 corre LOCAL (laptop Daniel). Cuando ejecuta stage PERSIST, llama Claude API una vez para traducir branches user-facing del JSON, guardando en `assigned_plans.plan_data_en`.
- Costo: ~$0.05 por plan, latencia 0 al cliente (asíncrono al authoring), determinístico.
- Fallback (planes legacy pre-feature donde `plan_data_en IS NULL`): job batch que traduce planes activos cuando un cliente flipea a `en` por primera vez; UI muestra "Translating your plan (~45s)" con skeleton. Sin esto, primera vista en-US de planes legacy es awful.
- **Voz**: system prompt del traductor exige "native US fitness coach copywriter, 2nd person 'you', motivational but not corny, NO British, NO Latin American calques". Output validado por linter `no-spanglish` + `no-british`.

**Supuesto a confirmar**: el JSON `plan_data` tiene secciones identificables como user-facing copy (descripciones, motivacionales, instrucciones) vs estructurales (reps, sets, exercise_id). Si la mezcla es indistinguible, Daniel debe etiquetar keys traducibles antes de Fase 3.

### Decisión 5 — Unidades imperiales | Confianza ALTA

- Datos en DB SIEMPRE métrico (kg, cm, km, °C). NUNCA convertir antes de guardar.
- Conversión en frontend vía composable `useUnits()` que envuelve `useLocale()` + `userStore.unitSystem`.
- API uniforme (no responde unidades distintas según user → no rompe cache).
- Métodos expuestos: `formatWeight(kg)`, `formatDistance(km)`, `formatTemperature(c)`, `parseWeightInput(value, unitHint)` (crítico: cliente escribe "180 lbs" → guarda 81.6 kg).

## 6. Los 3 modos de falla más probables en producción

### 1. ~~Vanilla PHP sirviendo página en español a cliente US~~ — DEUDA ACEPTADA (no bloqueador)
**Decisión Daniel 2026-05-17**: las rutas vanilla NO se tocan ni se migran. Si un cliente en-US navega a una ruta vanilla, verá copy en español. Trade-off explícito. Documentar en ADR-0004 y `README` del proyecto i18n. Se acepta porque migrar vanilla destruiría funcionalidad activa en producción.

### 2. Cache CDN/Redis envenenado mezclando locales
**Escenario**: endpoint API olvida `Vary: Accept-Language` o no incluye locale en la cache key. Cliente `es` ve respuesta `en` (o viceversa).
**Probabilidad**: MEDIA-ALTA (no hay convención actual ya que no había múltiples locales).
**Mitigación**: middleware global que añade `Vary: Cookie` y locale a toda cache key; smoke test post-deploy con doble-locale para cada endpoint nuevo o modificado; auditoría manual de cualquier `Cache::remember` en código.

### 3. Plan creado pre-feature, cliente cambia a en-US, ve `plan_data_en=NULL`
**Escenario**: cliente US flipea su locale a `en`, abre `/client/plan` — el plan fue creado antes del lanzamiento, no tiene traducción. Sin fallback bien diseñado: ve pantalla rota o se traduce caro on-demand cada vista.
**Probabilidad**: MEDIA (afecta sólo a clientes US existentes pre-launch).
**Mitigación**: job batch que pre-traduce planes activos a `en` cuando un cliente flipea a `en` por primera vez. UI muestra "Translating your plan (~45s)" con skeleton. Cachear en columna. Sin esto, primera vista en-US de planes legacy es awful.
