# 0004 — Internacionalización en-US: infraestructura técnica

**Fecha**: 2026-05-18
**Estado**: Fase 1 implementada (infraestructura). Fases 2-5 pendientes.
**Contexto**: ver auditoría completa en `docs/I18N_AUDIT_2026-05-17.md`.

WellCore está abriendo el portal cliente a usuarios native English speakers en EE.UU. La plataforma original es 100% es-CO. Esta ADR documenta la infraestructura técnica de i18n acordada con Daniel.

## Decisiones vinculantes

### 1. Stack: Laravel `lang/` + `vue-i18n@9` + endpoint API híbrido

- Backend: `lang/es/*.php` + `lang/en/*.php` (ya existen 20 archivos paralelos). Sirven Blade, Livewire, emails y validation messages.
- Frontend: `vue-i18n@9` montado en `app.js` con `legacy: false` (Composition API). Bootstrap inicial via meta + `window.__wcMessages` para evitar FOUC.
- Endpoint público: `GET /api/v/translations/{locale}` consolida todos los archivos PHP de `lang/{locale}/` en un JSON, cache Redis 1h, ETag para revalidación corta.
- App vanilla PHP (`wellcorefitness/`) NO se toca (ADR-0001). Si un cliente en-US navega a una ruta vanilla, ve copy en es-CO — **deuda técnica conocida y aceptada** por Daniel para preservar funcionalidad legacy.

### 2. Persistencia del locale

Migración aditiva (cumple ADR-0003) creada en `database/migrations/2026_05_18_120000_add_i18n_columns_to_clients_admins_assigned_plans.php`:

```sql
ALTER TABLE clients
  ADD COLUMN locale VARCHAR(10) NOT NULL DEFAULT 'es',
  ADD COLUMN locale_locked TINYINT(1) NOT NULL DEFAULT 0,
  ADD COLUMN unit_system VARCHAR(10) NOT NULL DEFAULT 'metric';

ALTER TABLE admins
  ADD COLUMN locale VARCHAR(10) NOT NULL DEFAULT 'es',
  ADD COLUMN locale_locked TINYINT(1) NOT NULL DEFAULT 0,
  ADD COLUMN currency VARCHAR(3) NOT NULL DEFAULT 'COP';

ALTER TABLE assigned_plans
  ADD COLUMN plan_data_en JSON NULL;
```

**VARCHAR(10) NO ENUM**: lección del incidente real `clients.plan` (ENUM en prod 2026-04). Permite agregar `pt-BR`, `es-MX`, `en-GB` sin migrar tipo. Validación en aplicación (FormRequest).

**Valores `'es'`/`'en'`** (no `'es-CO'`/`'en-US'`): alinea con `SetLocale` middleware existente y archivos `lang/es/`/`lang/en/`. La voz "Colombia neutro" vs "US native" es responsabilidad de copy, no de code de locale.

**Coach = `admins`**: no existe modelo `Coach` ni tabla `coaches`. Locale + currency + lock viven en `admins`. Daniel (superadmin) tendrá `locale_locked=1` desde seed para evitar accidentes.

**`assigned_plans.plan_data_en`**: el motor v2 traducirá branches user-facing del JSON al stage PERSIST (no on-read). Fase 3.

### 3. Resolución del locale (middleware `SetLocale`)

Prioridad fija:
1. Usuario autenticado con `locale` en columna → fuente de verdad.
2. Cookie `wc_locale` (1 año, `SameSite=Lax`, no encriptada — ya en allowlist de `bootstrap/app.php`).
3. Default `'es'`.

### 4. Lock (`locale_locked`)

- Backend (`MeController::updateLocale`): rechaza con **403** si `locale_locked=true`. Solo un admin puede sacar el lock desde panel admin (fuera de scope Fase 1 — sólo el flag está disponible).
- Frontend (`<x-language-switcher>` y composable `useLocale`): toggle deshabilitado + tooltip i18n.

### 5. Currency del coach

- `admins.currency` (default `'COP'`). Para coaches con `locale='en'` se setea `'USD'` (manualmente o en seed/admin UI futura).
- Aplica al dashboard coach (revenue, MRR, comisiones). El cliente NO se afecta — sus pagos siguen en COP a través de Wompi (la decisión de checkout USD se pospone).

### 6. Sistema de unidades (`clients.unit_system`)

- Default `'metric'` para es y `'imperial'` para en (derivar al crear cliente; editable manualmente).
- Storage en DB **siempre métrico** (kg, cm, km, °C). Conversión en frontend vía composable `useUnits()`.
- API uniforme para no romper cache: backend devuelve métrico siempre; el cliente formatea.

### 7. Contenido dinámico del motor v2 (Fase 3, futuro)

Opción C — bilingüe al PERSIST:
- Motor v2 corre LOCAL (laptop Daniel). Stage PERSIST llamará Claude API una vez por plan para traducir branches user-facing del JSON antes de INSERT.
- Costo: ~$0.05/plan, latencia 0 al cliente.
- Fallback (planes legacy con `plan_data_en IS NULL`): batch job + UI "Translating your plan (~45s)".

### 8. Coach asignado a cliente US

Debe ser **native English speaker**. Validación implícita: coach con `admins.locale='en'` + `currency='USD'`. Lógica de matching en lógica de asignación cliente↔coach (futura, ya fuera de Fase 1).

- ✅ Endpoint `GET /api/v/translations/{locale}` (cache Redis + ETag).
- ✅ Endpoint `PATCH /api/v/me/locale` con lock check.

- ✅ `resources/js/vue/i18n.js` (createI18n + lazy load).

## Riesgos vigentes

1. **Cache CDN/Redis envenenado mezclando locales** → middleware debe agregar `Vary: Cookie` y locale a cualquier cache key en endpoints autenticados. Pendiente verificar `Cache::remember` en código existente.
2. **Plan legacy en-US**: si un cliente flipea a `en` y su plan no tiene `plan_data_en`, hoy obtiene la versión en es (porque columna nueva queda NULL). Fase 3 resolverá con batch job.
3. **Vanilla PHP en español**: deuda aceptada §1.
